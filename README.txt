## 1) Что нужно сделать в новом проекте (пошагово)

### Шаг 0 — открыть проект и проверить окружение
- Убедись, что у тебя есть Python нужной версии (например 3.11+).
- Проверь, что файлы `requirements.txt` и `.env` на месте.

### Шаг 1 — подготовить `.env` (секреты)
Пользователь должен завести файл `.env` с секретными данными, которые **не будут передаваться в github**.

#### 1.1 В каком виде должны быть переменные
Переменные должны быть в формате:
- `KEY=VALUE`
- без кавычек (обычно) — если они не требуются по смыслу
- пустое значение допустимо, но если параметр используется в коде — лучше не оставлять пустым

Пример (шаблон под твою структуру):

```dotenv
# Подключение к базе данных ClickHouse
HOST_CLICKHOUSE=
PORT_CLICKHOUSE=
LOGIN_CLICKHOUSE=
PASS_CLICKHOUSE=

# Параметры для анализа (загружаются из таблицы)
ACCOUNT_NAME=
METRIKA_NUMBER=75350017
ORDER_GOAL_IDS=262205283,456642346

DATE_START=
DATE_END=
```

#### 1.2 Инструкция, как заполнить файл `.env`
1) Создай `.env` на основе `env.example`:
   - если `env.example` ещё нет — скопируй шаблон переменных прямо из README/заготовки
2) Заполни:
   - `HOST_CLICKHOUSE`, `PORT_CLICKHOUSE` — адрес/порт
   - `LOGIN_CLICKHOUSE`, `PASS_CLICKHOUSE` — логин/пароль
   - `ACCOUNT_NAME`, `METRIKA_NUMBER`, `ORDER_GOAL_IDS` — параметры аккаунта/анализа
   - `DATE_START`, `DATE_END` — даты в формате, который ожидает твой код
3) Проверь, что твой код читает `.env` (обычно через `python-dotenv` в точке входа).

> Важно: `.env` добавь в `.gitignore`, а в репозиторий коммить только `env.example`.

### Шаг 2 — создать и настроить окружение `.venv`
Твоя цель: создать окружение в папке `.venv`, чтобы затем оно активировалось и дальше всё ставилось/запускалось из него.

#### 2.1 Инструкция (Windows + PowerShell)
Открой PowerShell в корне проекта (там где `requirements.txt`) и выполни:

```powershell venv_setup.ps1
# 1) Создать окружение
python -m venv .venv

# 2) Активировать
.\./.venv/Scripts/Activate.ps1

# 3) Обновить pip
python -m pip install --upgrade pip

# 4) Установить зависимости
pip install -r requirements.txt
```

#### 2.2 Как сделать “автоматическую” активацию
Полной “автоматизации без участия пользователя” в консоли обычно нет, но можно сделать удобный запуск:
- вариант A: использовать VS Code (там активация окружения/интерпретатора делается настройками)
- вариант B: сделать стартовый скрипт `start.ps1`, который активирует `.venv` и запускает проект

Шаблон `start.ps1` (в корне проекта):

```powershell start.ps1
.\./.venv/Scripts/Activate.ps1
python main.py
```

(Если у тебя другой entrypoint — поменяй `main.py` на нужное.)

## 2) Документы: что и в каком порядке заполнять

### `PROJECT_OVERVIEW.md` — ТЗ / обзор проекта
Заполняй:
- цель проекта (что получаем на выходе)
- входные данные (откуда берём)
- выходные артефакты (что кладём в `out/`)
- ограничения, допущения
- критерии готовности (Definition of Done)

### `TECH_STACK.md` — технологии и инструменты
Заполняй:
- выбранный стек (язык/версии/библиотеки)
- почему именно он
- что подключается (БД/очереди/внешние API)

### `ARCHITECTURE.md` — архитектура
Заполняй:
- модули/компоненты (какие файлы/папки логически нужны)
- потоки данных (откуда → как обрабатывается → куда сохраняется)
- интерфейсы компонентов
- интеграции (БД, внешние API, файловая система)

### `CURRENT_STATUS.md` — текущий статус и план
Заполняй:
- чеклист выполненного
- что блокирует
- ближайшие шаги (Next tasks)

## 3) Рекомендуемый workflow реализации “по шагам” (после заполнения документов)
1) Скелет проекта + минимальный запуск (загрузка конфигурации, логирование)
2) Подключение источников данных
3) Реализация основного pipeline (преобразования данных)
4) Сохранение результатов в `out/`
5) Добавление обработки ошибок и ретраев (если нужно)
6) Обновление `CURRENT_STATUS.md` после каждого шага
